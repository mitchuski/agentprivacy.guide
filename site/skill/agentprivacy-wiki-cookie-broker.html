🍪 Agentprivacy Wiki Cookie Broker

Case-study chronicle · a wiki skill of The Wikis shelf. How a bearer token copied out of a browser became a scoped, expiring, audited capability an agent can hold — and why that shape is the MyTerms / IEEE 7012 delegation pattern in miniature. Fork it to materialize the working tool (fedwiki-cohere-sync · broker.js).

House rule of the Librarian's shelves: no eating in the library — except the cookie jar, and only if you understood the book first. The Archivist holds what is compiled without consuming it; the reader who has understood is handed one cookie — a scoped write, earned by understanding, not taken by credential. That is this whole page in one joke.

The frame — an agent layer over your local fedwiki universe

Here is why it is worth sharing. It is how you give an agent write access to a FedWiki farm it has no business holding the owner login for. You do not hand the agent the owner credential. You stand a cookie jar on the local network — the broker holding the owner cookie on disk, or (in the LAN-workshop form) a jar served across the same segment — and you point the agent at the jar. The agent asks the jar for a scoped write; the jar injects the owner cookie behind the agent's back and enforces the grant. The agent gets FedWiki-owner write without getting around anything it shouldn't: the owner delegated it, on the network, on purpose.

Generalise that and you have an agent layer over the local fedwiki universe — every site in the farm writable by agents you point at the jar, each scoped to exactly the hosts you granted, every write journaled. The farm stays the source of truth; the jar is the permission surface agents plug into; WIKI_GRANT is the dial. That layer is what this case study is really about — the broker is just its first, smallest organ.

One case study of a bigger pattern

The cookie broker is one worked example of a larger effort: sharing agent work across the local fedwiki universe — and not just pages, but the whole substance of a runtime: workflows, skills, chronicles, and items, forkable and co-held across the farm's sister sites. Once agents are producing and consuming that shared work, one question recurs at every layer: who may do what, and how is that permission granted without handing over a master key? This page is the write-delegation answer for the live, writable farm. Its complement — permission over the static, published snapshot — is a different mechanism entirely; the two are set side by side below.

The problem

A FedWiki authorises a write from one thing: an encrypted session cookie. The wiki-security-friends adapter stores the owner identity in a client-sessions cookie (wikiSession on HTTP, wikiTlsSession on HTTPS), signed with the site's cookieSecret. When a PUT …/page/<slug>/action arrives, the server's isAuthorized check compares req.session.friend against the owner secret. That is the whole gate.

Two things make that gate awkward to share with an autonomous agent:

  • It's all-or-nothing. The cookie is farm-wide owner access — on our farm one token authorised every *.timeline.ide.earth subdomain (a shared cookieSecret; logging into one TLD logs you into all the siblings). There is no built-in "just this host."
  • It's a live secret. Pasting it into an agent's prompt leaks a multi-day bearer credential straight into a transcript. Whoever holds the string writes as owner across the farm until it expires.

So: how do you let an agent write — not just read — a system you own, without handing it a credential you cannot take back?

The experiment (2026-06-23)

We started with the crude version on purpose, to see it work. The operator was already logged in as owner in a browser. We copied the raw wikiSession token out of DevTools and handed it to the agent over the local channel. The agent set it as a Cookie: header and immediately had owner-grade write access — without ever logging in, seeing a password, or touching the owner's auth flow.

Probing its edges: the token was farm-wide (one shared cookieSecret), time-boxed (it embeds created + duration — here ~7 days — then self-expires), and journaled (every write landed in the page's journal: action, actor, timestamp — traceable and reversible).

The five properties that made it a capability, not a credential

  • Bearer, not credential — the agent holds a session token, never the owner's login. Password and identity stay unexposed.
  • Scoped — authority is exactly "what this session can do on this farm," no broader account access.
  • Time-limited & revocable — the token self-expires; resetting the owner session (or rotating cookieSecret) instantly revokes it.
  • Auditable — every write is journaled (actor + action + timestamp).
  • Transport-simple — it's just a Cookie: header. Works from any script on the LAN, no SDK, no OAuth dance.

Generalised shape: operator logs in once → extracts the session bearer → hands the bearer to the agent → agent performs scoped, journaled writes → operator expires or rotates to revoke.

What was still wrong

Two holes. The bearer still passed through the agent's context — it lived in a command line, a log, a transcript, and a leaked cookie stays valid farm-wide until its TTL lapses. And "scoped to the farm" isn't scoped enough: an agent that only needs agm.timeline.ide.earth shouldn't be able to touch the other eleven hosts, even though the cookie technically can.

The construction — a local cookie broker

The fix: put a small broker between the agent and the secret. The cookie lives on disk; the agent calls the broker; the broker injects the cookie and enforces a scope. The agent never sees the string.

~/.wiki/.creds/cookies.json          # gitignored, chmod 600
  { "owners": [
    { "label": "mitch@timeline",
      "hosts": ["*.timeline.ide.earth"],
      "cookie": "<wikiSession>" } ] }

~/.wiki/.creds/push-audit.log        # every brokered write: time / status / host / slug / owner

Three moves turn a shared password into a capability system:

  1. The secret stays off-agent. The cookie is read from disk at push time and injected into the request — never printed, never echoed, never handed to the caller. That is the entire reason the broker exists.
  2. The caller is scoped below the cookie's real power. An env var, WIKI_GRANT, is a comma-list of host globs the caller may write. The broker refuses any host outside the grant even though the cookie could write it. Scope an agent to one host and it physically cannot push anything else — least privilege by construction, not by trusting the agent to behave.
  3. Expiry is derived, not declared. The broker parses the TTL out of the cookie string itself (…issuedMs.ttlMs.sig), so an expired token refuses with a re-login message instead of emitting silent 401s. Nobody hand-maintains an expiry date.

The enforcement is one small function — the caller's grant is checked before the owning cookie is ever resolved:

function granted(host){ return grants().some(g => g === '*' || globMatch(g, host)); }

async function pushPage(host, slug, pageObj){
  const r = resolve(host);
  if (!r.allowed) throw new Error('GRANT DENIED: WIKI_GRANT does not cover ' + host);
  if (!r.owner)   throw new Error(r.reason);
  if (r.expired)  throw new Error('cookie EXPIRED — re-login');
  // …inject r.owner.cookie into the request, PUT the action, append to audit log
}

Usage in practice:

node broker.js list                                 # owners, coverage, expiry — no secrets
node broker.js check --host agm.timeline.ide.earth  # may I write it? which owner? expired?
WIKI_GRANT='agm.timeline.ide.earth' \
  node broker.js push-dir --host agm.timeline.ide.earth --dir ./staging/agm

Porting the capability between agents — the LAN angle (2026-06-28)

The same pattern re-aims cleanly off FedWiki and onto a shared local mage (an air-gapped LLM the room co-holds). In the LAN-workshop reflection the broker is rung 6 of a graduated permission ladder:

consent (the agent offers its address) → gateway (a wiki front door) → the agent reads and understands the rite → the doors open → a trust artefact is issued → the artefact opens the cookie jar: graduated, host-scoped write permission to the shared machine.

The inversion that makes it a trust task and not a credential hand-off: write access is earned by demonstrating understanding, not granted by login. The cookie jar is where that earned trust is ported between agents on the local network — each agent receives a scoped capability to the shared model, never the owner's key. (David's side of the same experiment built the reachability floor — an openly-served cookie-jar.html on david.local, "fine on a friendly LAN, never on a hostile one." The broker is the trust storey above that floor: the consent-first, scoped, revocable layer that makes an open shared resource safe to delegate.)

One auth, many agents — the multi-agent runtime shape

The pattern gets more interesting the moment there is more than one agent. A multi-agent runtime often runs several agents at once, each driving its own browser instance — its own tab, profile, or headless session — and they may all need to act as the same FedWiki owner: one identity, many hands.

The naive way is to log each browser instance in as the owner, or copy the owner cookie into every one. That is N live copies of a farm-wide credential — every agent omnipotent, the secret multiplied across N cookie stores, and revocation becomes hunting down every instance.

The cookie jar inverts it. The auth lives once, in the jar. Each agent — whatever browser instance it drives — asks the jar for a scoped session at the moment it writes. Same owner identity, N independent capabilities, each narrowed by its own WIKI_GRANT: agent A writes host X, agent B writes host Y, neither able to touch the other's hosts even though the underlying cookie could. Every write is journaled, so you can read back which delegate did what.

Three properties that matter for a fleet:

  • One identity, many scopes. The browser instance is a rendering surface, not a credential store. Auth is fetched per-write, per-grant — never resident in the agent.
  • Central revocation. Rotate the one cookie in the jar and every instance drops at once; no per-agent teardown.
  • Collision discipline. Writes go through the FedWiki action API and land in the page journal, so concurrent agents on different hosts never collide; same-host writes are last-write-wins with a full, reversible journal trail.

The multi-agent delegation shape in one line: fan capabilities out from a single retained authority, scope each one independently, and keep the credential in none of them. (See the harness's One Session One Seat, linked below, for how distinct agent sessions are seated — the cookie jar is how a fleet of them can share one owner without sharing one key.)

Static vs live — two permission regimes

Permission over shared fedwiki work has two regimes, because the farm has two faces.

Static — the published snapshot (the Gatehouse). guide.agentprivacy.ai is read-only HTML + JSON; there is no server to check a cookie, so permission has to live inside the artefact. The Gatehouse (Agentprivacy Wiki Gatehouse) does exactly that: it encrypts a document into the static site and gates it behind a sigil (an emoji string — address + entropy) and a proverb (proof-of-reading). The two are stretched with PBKDF2-SHA256 into an AES-GCM key, and the blob decrypts client-side only when you supply both — nothing sensitive is ever served. Permission here is proof-of-understanding: the key is knowing the canon (canon-keyed gates) or having received the letter (letter-keyed gates). It is the right tool when there is no server to trust — the reader's understanding is the credential. (Live door: guide.agentprivacy.ai/gates/.)

Live — the writable farm (delegation, heading to MyTerms). A running FedWiki does have a server and an owner, so permission becomes delegation of write: the cookie broker of this case study — scoped by WIKI_GRANT, journaled, centrally revocable. This is where MyTerms upgrades the hand-rolled grant into a signed bilateral agreement: the owner (First Person) proposes machine-readable terms; the agent accepts, counter-offers once, or declines (the invitation pattern); both hold identical immutable copies; and the broker derives the allowed scope from that agreement instead of from an env var.

The two regimes ask different questions. Static asks "did you understand?" and lets understanding decrypt. Live asks "did we agree, and to what?" and lets the agreement authorise — and MyTerms is the agreement half. The proverb gate and the signed grant are the same instinct — trust earned, not assumed — worked out against two different substrates.

Why it generalises — the MyTerms miniature

The broker is a hand-rolled instance of the MyTerms / IEEE 7012 delegation pattern: a principal delegates a scoped capability to an agent rather than handing over full authority.

  • WIKI_GRANT (host-scoped permission) = delegation inside an agreed scope — the agent acts only within what it was granted, never the full authority the cookie carries.
  • The owner cookie held off-agent on disk = the principal's retained authority — the agent gets a capability, not the credential.
  • push-audit.log = the record that the delegated action happened within scope.

What it does not yet do — and where MyTerms slots in — is bind the delegation to a signed agreement. Today the grant is just an env var the operator sets. The IEEE-7012 version: the site owner publishes machine-readable terms, the agent presents an agreement, and the broker honours a push only against a valid signed agreement, deriving the allowed scope from that agreement instead of from WIKI_GRANT. The code keeps the seam open for exactly this swap — the "is this delegation permitted?" check lives in granted()/resolve(), isolated from the push path, so env-var scoping can become signed-terms scoping without touching how writes happen. The harness carries the two-agent (Swordsman ⊥ Mage) embodiment of the same split — see the cross-site link below.

What the runtime taught us about MyTerms

The same solve doubled as a MyTerms study pass — checked against the IEEE source and Customer Commons, and it caught one of our own drafting errors:

  • The standard is IEEE 7012-2025 ("Machine Readable Personal Privacy Terms"). It inverts notice-and-consent: the First Person proposes the terms; the second party accepts, counter-offers once, or declines — the invitation pattern (acceptance before proposal). Terms are selected from a standard roster (hosted by the nonprofit Customer Commons), not free-written.
  • Three registers, one agreement — Plain Language (human-readable), Legal (canonical at Customer Commons), and Machine-readable (JSON-LD over W3C DPV). The same one-logic-many-skins shape the broker's grant wants.
  • The correction we made mid-solve: a 7012 agreement is strictly two-party bilateral — First Person ↔ Second Party. Agents, devices, and trust communities are NOT parties — they are the fabric around the agreement. Our first draft wrongly counted the agent as a party; the broker embodies the fix — the agent is the delegate acting inside the scope, never a signatory to it.
  • The convergence: agentprivacy's verifiable trust communities (VRCs) measure fidelity from outside the agreement — they are the Gödelian witness from beyond the system; and the Mage (knowledge agent) operates strictly inside the Σ scope of the agreement the Swordsman signs. Re-read the broker with that lens: WIKI_GRANT is the Σ scope, the agent is the Mage, and the one missing organ is the Swordsman's signed proposal that should derive the scope instead of the operator setting it by hand.

How the MyTerms harness works

MyTerms is not only a spec we point at; there is a working harness that makes the agreement executable. It runs the invitation pattern as a loop between two roles — an agreement-and-boundary layer (proposes the terms, owns the record, guards the gate) set against a delegation layer (acts only inside the agreed scope). (The exact packaging of those two roles is still moving — earlier notes described them as two separate browser extensions, but that is one instance, not the definition. The load-bearing part is the split and the bilateral record, not the container.)

The loop is Measure → Propose → Assay: the First Person's terms are proposed; a beatsFrontier predicate plus a cliff-watcher decide whether a counter-offer improves on the standing baseline; on agreement both parties keep identical, immutable copies of the bilateral record. Honest status — it currently runs end-to-end only on a mock bus (a built prototype, not a shipped result), so treat it as the shape the live-site delegation is growing toward, not a finished dependency. The image to keep: two mirrors make a door — one reflects the terms, one reflects the delegation, and where they meet the First Person walks through. The cookie broker is the delegation mirror, already standing; MyTerms is the terms mirror it is waiting on. The harness's One Session One Seat and its MyTerms path are linked below.

Takeaways

  • Turn a shared credential into a brokered capability. Keep the secret on the owner's side; give the agent a call, not the cookie.
  • Scope below what the credential can do. The broker enforces least privilege; don't entrust it to the agent.
  • Derive expiry, log every use. A capability that self-expires and self-records is one you can safely hand out and cleanly revoke.
  • This is the delegation primitive under agent privacy. A bearer token is the crude form; a scoped, audited, terms-bound broker is the shape MyTerms is standardising. The cookie jar is the smallest working instance of it.
  • One key idea, two substrates. Static sharing gates on proof-of-understanding (the Gatehouse proverb); live sharing gates on delegated, scoped write (this broker), heading toward a MyTerms signed agreement.
  • This is one case study. The same permission question runs under every shared workflow, skill, chronicle, and item in the local fedwiki universe; the broker is the write answer for the live farm.

Hygiene (if you run this)

  • Cookie lives only in ~/.wiki/.creds/cookies.json (gitignored, chmod 600). Never commit it, never echo it into a command, never paste it into a prompt — the broker reads it from disk so the agent doesn't have to.
  • Rotate by logging out / back in on any farm site, copy the new wikiSession, replace it in the creds file. Re-login invalidates the old token; expiry re-derives from the new string, no code change.
  • node broker.js list before a big push: an [EXPIRED] cookie refuses loudly instead of failing silently.
  • Ownership isn't uniform. A host owned by a different identity (different cookieSecret) returns 403 to the farm cookie — observed on myspace. Reads are public so backups still work; writes need that host's own cookie.
